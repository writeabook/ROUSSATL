# 对象生命周期与所有权

本文档定义了所有公共 OSAL 对象的所有权、生命周期和资源管理规则。
每个后端必须遵守这些规则，除非明确记录为 `Unsupported`。


## 1. 目的

本章节定义了所有公共 OSAL 对象的所有权、生命周期和资源管理规则。

API trait 定义了对象**能做什么**。本章节定义了**对象是什么**——所有权、
身份标识、清理行为和句柄语义。每个后端必须遵守这些规则，除非明确记录
为 `Unsupported`。

## 2. 后端持有的对象

所有公共 OSAL 对象都是**后端持有的句柄**。公共 API 暴露不透明类型；
其内部表示是后端特定的，**不属于**公共契约。

```
Mutex<T>
  ├── POSIX:    pthread_mutex_t
  ├── FreeRTOS: SemaphoreHandle_t
  └── Mock:     MockMutexState

Queue
  ├── POSIX:    pthread_mutex_t + condvar 对 + 环形缓冲区
  ├── FreeRTOS: QueueHandle_t
  └── Mock:     MockQueueState
```

同样的模式适用于 `Task`、`Timer` 和 `Semaphore`。
后端原生句柄**绝不得**出现在公共 API 中。

## 3. 所有权

每个公共对象恰好拥有一个后端资源。

除非另有规定：
- 所有权是**唯一的**（不与其他公共对象共享）。
- 资源销毁通过 RAII 是**确定性的**。
- 公共对象被 drop 时释放后端资源。

## 4. 共享所有权

某些对象类型可以通过 `Clone` 支持共享所有权。克隆**不会**复制底层
操作系统资源。相反，克隆创建另一个引用**同一**后端对象的句柄。

```
Queue
 ├── Handle A ──┐
 └── Handle B ──┤
                ▼
         同一个后端队列
```

对象类型是否可克隆是其 API 契约的一部分。未明确记录为可克隆的类型
不得被克隆。

## 5. Drop 语义

丢弃**最后一个**对象句柄会释放后端资源。丢弃克隆句柄仅递减引用
计数（参见 [14](#14-引用计数)）。Drop 操作**不得**无限期阻塞。

| 对象 | 必需的 Drop 行为 |
|------|-----------------|
| Mutex | 销毁后端 mutex |
| Semaphore | 销毁后端 semaphore |
| Queue | 释放队列资源 |
| Timer | 停止定时器并释放资源 |
| Task | 仅释放句柄（不得隐式 join） |

阻塞清理操作应通过**显式 API**（`join`、`close`、`stop`）暴露，
而非通过 `Drop`。

## 6. Task 生命周期

`Task` 句柄表示后端任务对象的所有权。

- `join()` 等待任务完成。
- 如果 `join()` 返回 `Timeout`，任务句柄保持有效并可重新使用。
- 成功 `join()` 后缓存退出码；后续调用立即返回该值。
- 丢弃 `Task` 句柄**绝不得**无限期阻塞。

## 7. Queue 生命周期

`Queue` 在显式关闭或丢弃之前一直存在。

关闭队列应：
- 拒绝未来的 `send` 操作（始终返回 `Error::QueueClosed`）。
- 唤醒所有被阻塞的发送者（它们返回 `Error::QueueClosed`）。
- **仅在队列为空时**唤醒被阻塞的接收者；在非空队列上阻塞的接收者
  继续排空（drain）。
- 保留已入队的消息——它们通过 `recv` 保持可读，直到队列为空或
  被丢弃。

Close 之后，`recv` 仅在队列已关闭**且**为空时返回
`Error::QueueClosed`。

## 8. Timer 生命周期

丢弃定时器应阻止任何未来的回调。

后端应保证在定时器成功销毁后，不会有回调开始执行。如果在销毁时
回调已经在执行中，则适用后端特定的同步规则。

## 9. 后端独立性

应用代码绝不得依赖后端原生对象类型。以下属于**实现细节**：

- `pthread_mutex_t`
- `pthread_t`
- `SemaphoreHandle_t`
- `TaskHandle_t`
- `QueueHandle_t`

后端可以自由选择任何内部表示，只要外部可观察行为符合本契约。

## 10. 资源映射

后端实现可以将公共对象以不同方式映射到原生资源。例如：

```
POSIX Queue              FreeRTOS Queue
    ↓                         ↓
pthread mutex             xQueueHandle
pthread condvar
ring buffer
```

只要外部可观察语义保持一致，不同的内部实现是可以接受的。

## 11. 一致性要求

一致性测试仅验证**可观察行为**。它们不得假设：

- 特定的后端数据结构。
- 原生句柄类型。
- 内存布局。
- 实现算法。

只有外部可观察语义属于本契约的范围。

## 12. 对象身份

每个公共对象恰好代表**一个**后端资源。

- 两个克隆的句柄引用**同一个**后端对象。
- 两个独立创建的对象始终代表**不同**的后端资源。
- 后端在克隆期间**不得**复制原生对象。

```
let q2 = q1.clone();

q1 ──┐
     ├── 同一个 Queue（而非两个独立的队列）
q2 ──┘
```

此规则是强制性的。没有它，Mock、POSIX 和 FreeRTOS 后端将在克隆
语义上产生分歧，破坏可移植性保证。

## 13. 句柄相等性

引用同一后端对象的两个句柄必须比较相等。

```rust
let q2 = q1.clone();
assert_eq!(q1.handle(), q2.handle());
```

独立创建的对象句柄必须比较不等：

```rust
let q1 = Queue::new(8, 4)?;
let q2 = Queue::new(8, 4)?;
assert_ne!(q1.handle(), q2.handle());
```

## 14. 引用计数

当类型支持 `Clone` 时，所有句柄平等地对引用计数做贡献。后端资源
仅在**最后一个**句柄被丢弃时才被销毁。

```
Queue::new()  →  refcount = 1
  q2 = q1.clone()  →  refcount = 2
  drop(q1)         →  refcount = 1  （资源仍然存活）
  drop(q2)         →  refcount = 0  （资源被销毁）
```

所有句柄地位平等——不存在"主"句柄。任何句柄都可以执行对象 API
允许的任何操作。

## 15. 最后一个句柄丢弃时的行为

显式 `close()` 或 `delete` 操作**必须**唤醒所有阻塞的任务，并返回
适当的错误（`Error::QueueClosed`、`Error::LockFailed` 等）。

最后一个句柄的 `Drop` 仅负责**资源清理**。不要求它唤醒阻塞的任务。

实际上，当公共 API 使用 `&self` 方法进行阻塞操作时：

```rust
queue.recv(&mut buf, Timeout::Forever)?;
```

阻塞的调用者持有该句柄的共享引用。在有任务阻塞时引用计数不可能
归零，因此在进程内后端（POSIX、Mock）中，最后一个句柄的 drop 不
可能与阻塞调用同时发生。这由 Rust 借用模型保证。

对于使用 FFI 的后端（FreeRTOS），如果阻塞调用在内部使用裸指针，
实现必须确保在另一个任务显式关闭或删除资源时，先唤醒后释放的顺序
得到维护。

## 16. 共享句柄的显式关闭

当在一个句柄上调用 `close()` 时，所有句柄受到同等影响。

```rust
let q1 = Queue::new(8, 4)?;
let q2 = q1.clone();

q1.close();

// 两个句柄都看到队列已关闭：
q1.send(&data, Timeout::NoWait)?; // Error::QueueClosed
q2.send(&data, Timeout::NoWait)?; // Error::QueueClosed
```

`close()` 之后，内省方法（`len()`、`capacity()`、`is_empty()`、
`is_full()`、`msg_size()`）继续工作并反映关闭时的状态。

Close 是**幂等的**：多次调用 `close()`（包括从不同句柄调用）是安全
的，除第一次调用外没有额外效果。

## 17. 销毁期间的信号安全性

后端必须确保关闭或丢弃共享资源不会使任务处于不一致状态。

具体要求：

1. **先唤醒后释放**：被阻塞的任务必须在原生资源被释放之前从等待
   队列中移除。
2. **禁止 use-after-free**：一旦被阻塞的任务以错误被唤醒，它不得
   通过被唤醒的路径访问原生资源。
3. **回调安全性（Timer）**：`stop()` 或 `Drop` 之后，定时器回调
   不得被调用。如果回调已在执行中，后端必须确保它在不访问已释放
   内存的情况下完成（例如通过引用计数或代次计数）。
4. **关闭-排空顺序（Queue）**：`close()` 时已在队列中的消息保持
   对 `recv()` 可用，直到队列为空或被丢弃。Close 仅阻止新的发送；
   它不会丢弃现有消息。
